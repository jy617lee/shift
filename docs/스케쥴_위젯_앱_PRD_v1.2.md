# PRD: 시프트 근무자용 스케쥴 위젯 앱 (MVP)

버전 1.2 · 2026-06-11

---

## 1. 배경 및 문제

교대(시프트) 근무자는 자신의 근무 시간을 확인하려면 매번 사내 스케쥴 앱에 로그인하고 메뉴를 찾아 들어가야 한다. 주 단위로 출퇴근 시간이 달라지는 근무 특성상 확인 빈도는 높은데 접근 비용이 크다.

본 앱은 사용자가 사내 앱의 주간 스케쥴 화면을 **스크린샷 1장으로 등록**하면, 홈 화면 **위젯**에서 오늘의 근무 시간을 즉시 확인할 수 있게 한다.

- **초기 타겟**: 국내 특정 프랜차이즈 카페 직원 (홍보 채널에서만 타겟팅하며, 앱 명칭·스토어 설명·앱 내 어디에도 해당 상호를 사용하지 않는다)
- **포지셔닝**: 범용 "교대근무 스케쥴 위젯" 앱
- **향후 확장**: 타 스케쥴표 포맷(템플릿) 추가 → 모든 시프트 근무자 대상

## 2. MVP 범위

| 포함 | 제외 (향후) |
|---|---|
| 스크린샷 기반 주간 스케쥴 등록 | 함께 일하는 동료 표시 위젯 |
| 파싱 결과 확인·수정 화면 | 캘린더(OS) 연동 |
| 홈 위젯 3종 (2×1, 4×1, 4×2 주간 미리보기) | 하단 배너 광고 |
| 인식 실패 이미지 옵트인 전송 (개발팀 슬랙) | 해외 포맷 지원 |
| 설정 화면 (확인 생략 옵션 재설정, 앱 정보) | 수동 스케쥴 입력(이미지 없이) |
| | 홈 화면에서의 스케쥴 수정 (등록 후 변경은 재등록으로 처리) |
| | 위젯 2×2 (MVP 이후 추가 예정) |

## 3. 핵심 가정 및 제약

- 스케쥴표 포맷은 단일 템플릿(첨부 예시 이미지)으로 고정. 주간 단위(월~일), 본인 1인의 스케쥴만 표시됨.
- 자정을 넘기는 근무는 없다. 따라서 `퇴근시간 ≤ 출근시간`이면 파싱 오류로 처리한다.
- 이미지는 인식의 입력으로만 사용하고 **파싱 완료 즉시 폐기**한다. 서버는 물론 기기에도 보관하지 않는다. 유일한 예외는 §6.3의 사용자 동의 기반 실패 이미지 전송(전송 완료 후 로컬 폐기).
- 모든 데이터는 로컬 저장. 기기 변경·앱 삭제 시 데이터는 복구되지 않는다 (온보딩 또는 설정에 1줄 고지).
- 플랫폼: Android / iOS (위젯 그리드 명칭은 Android 기준, iOS는 Small/Medium 대응표 §7 참조).

## 4. 화면 플로우

### 4.1 앱 첫 페이지 (홈)

- 주 단위 스케쥴 **표**를 세로로 이어 붙인 리스트. 앱 진입 시 **오늘 날짜 행이 화면 상단**에 오도록 스크롤 위치 고정.
- 오늘 날짜 행은 **하이라이트 색 테두리**로 표시.
- **상하 스크롤**: 위로 스크롤하면 지난주, 아래로 스크롤하면 다음주. 각 주는 날짜 범위 헤더(예: 6/22~6/28)가 있는 섹션.
- 스케쥴이 등록되지 않은 주는 빈 섹션 + "스케쥴 추가" 유도 문구.
- 우하단 플로팅 **[스케쥴 추가]** 버튼.

### 4.2 스케쥴 등록 플로우

1. [스케쥴 추가] 탭 → OS 갤러리(포토 피커) 오픈.
2. 사진 선택 → "등록 중" 인디케이터 표시 (텍스트 + 애니메이션 아이콘).
3. **1차 검증 — 스케쥴표 여부 판별** (§6.1)
   - 실패: 얼럿 `"지원하지 않는 이미지 형식입니다. 스케쥴 화면을 캡처해 다시 시도해주세요."` → 종료. 이미지는 어디에도 저장·전송하지 않음.
4. **2차 처리 — 데이터 추출** (§6.2)
   - **3행 이상 매핑 실패** 시: 다이얼로그 `"현재 지원하지 않는 스케쥴표 형식입니다. 이 이미지를 보내주시면 빠르게 지원하겠습니다. 보내시겠어요? (이미지에 이름·사번이 포함될 수 있습니다)"` → [보내기] / [취소]
     - [보내기]: 스케쥴 영역만 크롭한 이미지를 서버 업로드 → 개발팀 슬랙 전송 (§6.3) → 로컬 이미지 폐기
     - [취소]: 아무것도 저장·전송하지 않고 종료.
   - **1~2행 매핑 실패** 시: 실패가 아닌 부분 성공으로 처리하고 확인 화면으로 진행.
5. **확인 화면**:
   - 화면 상단에 **원본 이미지(스케쥴 영역)**, 하단에 추출된 7일치 데이터 표(날짜 / 출근 / 퇴근 / 근태코드)를 함께 표시하여 대조 가능하게 한다.
   - 각 칸은 탭하여 수정 가능. 시간 입력은 **숫자 키패드**(HHMM 4자리 입력 → 자동 콜론).
   - 매핑 실패 행은 하이라이트 + 빈 입력칸으로 표시하고 `"인식하지 못한 날이에요. 직접 입력해주세요."` 안내. 미입력 상태로도 등록은 가능(해당일은 미등록일 취급).
   - **[앞으로는 확인하지 않기]** 옵션: 활성화하면 이후 등록에서 정상 추출 시 확인 화면을 생략하고 즉시 등록(완료 토스트에 요약 표시). 단, **매핑 실패 행이 1개라도 있으면 이 설정과 무관하게 확인 화면을 표시**한다. 설정 화면(§4.4)에서 언제든 재활성화 가능.
6. 등록 완료 토스트 + 홈으로 복귀. **원본 이미지는 이 시점에 폐기.**
7. 같은 주에 기존 스케쥴이 있으면: `"6/22~6/28 스케쥴이 이미 있습니다. 새 데이터로 교체할까요?"` → 교체 / 취소.

> 확인 화면(5)은 MVP 기본 동작이다. OCR 오류가 위젯에 그대로 노출되면 제품 신뢰가 무너지므로 기본적으로 사용자가 최종 데이터를 확인하되, 정형 포맷 특성상 정확도가 높으므로 사용자가 원하면 생략할 수 있게 한다(매핑 실패 시에는 항상 표시).

### 4.4 설정 화면

앱 내 "설정" 탭에서 접근. 홈 화면 우상단 아이콘 또는 내비게이션으로 진입.

**등록 섹션**
- **OCR 결과 확인 건너뛰기**: 토글. 활성화 시 §4.2(5)의 확인 화면을 생략하고 즉시 등록. 매핑 실패 행이 있는 경우에는 이 설정과 무관하게 확인 화면을 표시. 기본값 OFF.

**앱 정보 섹션**
- 버전 정보 표시 (읽기 전용).

## 5. 데이터 모델

```
ScheduleWeek
  - week_start_date (date, PK)        # 월요일 날짜, 연도 포함
  - days: ScheduleDay[7]

ScheduleDay
  - date (date)
  - type: WORK | OFF | OTHER | UNREGISTERED
  - start_time, end_time (nullable, type=WORK일 때 필수)
  - code_label (string)               # "정상", "정규휴일", "업무외상병" 등 원문 그대로
  - source: PARSED | EDITED | MANUAL  # OCR 결과 / 사용자 수정 / 인식 실패 후 직접 입력
```

이미지 파일은 보관하지 않으므로 데이터 모델에 포함되지 않는다.

## 6. 인식 파이프라인

온디바이스 처리 원칙. 커스텀 ML 모델을 탑재하지 않고 **OS 기본 OCR(Android: ML Kit Text Recognition / iOS: Vision Framework) + 규칙 기반 파서**를 사용한다. 파서는 템플릿 플러그인 구조로 작성한다(향후 포맷 추가 = 템플릿 추가).

### 6.1 1차 검증 (스케쥴표 여부)

OCR 결과에서 다음을 모두 만족하면 통과:
- 헤더 키워드 검출: "출근", "퇴근", "근태코드" (또는 "주간 스케줄" 타이틀)
- 날짜 패턴 `MM/DD(요일)` 3개 이상 검출

미달 시 → 지원하지 않는 포맷 얼럿 (전송 제안 없음).

### 6.2 2차 처리 (데이터 추출)

- 행 단위로 날짜 ↔ 출근/퇴근/근태코드를 매핑. 7행 기대.
- **계획/실적 구분**: P(계획) 값을 기준으로 추출. R(실적) 값이 함께 있는 경우에도 P를 사용.
- **근태코드 처리**: "정상" → WORK(시간 필수). 시간 셀이 비어 있고 코드/텍스트만 있는 행(정규휴일, 업무외상병, 연차 등) → OFF/OTHER, `code_label`에 원문 그대로 저장하고 위젯·홈에 그대로 표시. 알 수 없는 코드를 임의 해석하지 않는다.
- **연도 부여 규칙**: 이미지에는 연도가 없으므로, `오늘 기준 −1주 ~ +8주` 범위에 들어오는 연도를 부여한다. 어떤 연도로도 범위에 들어오지 않으면 추출 실패로 처리한다. (연말연초 경계 자동 해결)
- **무결성 검증**: WORK 행에서 `퇴근 ≤ 출근`이면 해당 행 매핑 실패.
- **"매핑 실패"의 정의**: 날짜는 검출되었으나 (a) 시간도 근태 텍스트도 추출되지 않았거나, (b) 추출값이 무결성 검증에 걸린 행. **시간 없이 근태 텍스트만 있는 휴무 계열 행은 정상 매핑이며 실패가 아니다.**
- **부분 실패 처리**: 7행 중 매핑 실패가 **1~2행이면 부분 성공** — 확인 화면에서 해당 행을 하이라이트하고 사용자가 직접 입력. **3행 이상이면 추출 실패** — 포맷 자체가 다를 가능성이 높으므로 §4.2-4의 전송 제안 다이얼로그로 분기.

### 6.3 실패 이미지 전송 (옵트인)

- 트리거: 1차 통과 + 2차에서 3행 이상 매핑 실패 시에만.
- 사용자가 [보내기]를 누른 경우에 한해, 스케쥴 테이블 영역만 **크롭**한 이미지(상태바·내비게이션 제거)를 서버에 업로드하고 개발팀 슬랙 웹훅으로 전달. 전송 완료 후 로컬에서 폐기. 메타데이터는 앱 버전, OS, 실패 행 수만 포함 (사용자 식별자 미포함).
- 개인정보처리방침에 수집 항목("스케쥴표 이미지 — 이름·사번 포함 가능"), 목적(인식 개선), 보관 기간(**서버 보관 30일 후 자동 파기**) 명시. 스토어 개인정보 라벨에도 동일 반영.

> **구현 상태**: 현재 `FailedImageReporter`는 NoOp 구현체(`NoOpFailedImageReporter`)로 대체되어 있음. 실제 서버 업로드는 별도 이슈로 구현 예정.

## 7. 위젯 스펙 (4종)

공통:
- 탭 시 앱 첫 페이지로 이동.
- 데이터 소스는 당일 `ScheduleDay`.
- **휴무일(OFF/OTHER)**: 날짜 + `code_label`(예: "정규휴일")만 표시. 근무시간 영역 비움.
- **미등록일**: 날짜 + "스케쥴 없음" 표시.
- **갱신 정책**: 매일 00:00 타임라인 갱신 + 스케쥴 등록/수정 직후 즉시 갱신.

| 위젯 | 레이아웃 | 비고 |
|---|---|---|
| 2×1 | 우측: 날짜(11) / 좌상단: 요일(월) / 좌하단: 근무시간(7:30–13:30) | iOS Small 대응 |
| 4×1 | 좌: 날짜(6/11 목) / 우: 근무시간 또는 근태코드 | iOS Medium 대응 |
| 4×2 주간 미리보기 | 상단 헤더: 오늘 날짜·요일 + 오늘 근무시간(또는 근태코드) / 파도형 구분선 / 하단 그리드: 오늘 ±2일(5칸) 날짜·근무시간 | iOS Medium 대응 |
| 2×2 | MVP 이후 추가 예정 | — |

> **2×1·4×1 공통**: 휴무일은 근태코드만 표시, 미등록일은 "-" 표시.

**4×2 주간 미리보기 상세**
- 헤더(상단): 요일·날짜 + 오늘의 근무시간(또는 `code_label` / 미등록 시 "-").
- 파도형 구분선: 헤더와 그리드를 시각적으로 분리.
- 주간 그리드(하단): 오늘 기준 −2일 ~ +2일, 총 5칸. 각 칸에 요일·날짜·근무시간(또는 상태) 표시. 오늘 칸 강조.
- 갱신 정책: 매일 00:00 타임라인 갱신 + 스케쥴 등록/수정 직후 즉시 갱신.

## 8. 비기능 요구사항

- 등록 플로우(선택→확인 화면)는 일반 기기에서 3초 이내.
- 인식 파이프라인은 완전 오프라인 동작 (전송 기능만 네트워크 필요).
- 예시 포맷 기준 추출 정확도 목표: 행 단위 98%+ (내부 테스트셋 50장 기준).

## 9. 성공 지표 (MVP)

- 활성 사용자 중 위젯 설치율 ≥ 60%
- 주간 스케쥴 재등록률(리텐션 프록시) ≥ 50%
- 등록 시도 대비 추출 성공률 ≥ 90% (실패 이미지 수집으로 추적)

## 10. 결정 이력

1. 퇴근 후 위젯: 다음 근무 미리보기 채택 (없으면 "오늘 근무 종료").
2. 확인 화면: 사진+추출표 병행 표시, 숫자 키패드 입력, "앞으로 확인하지 않기" 옵션(매핑 실패 시 강제 표시).
3. 실패 이미지 서버 보관: 30일 후 자동 파기.
4. 홈: 상하 스크롤, 오늘 행 상단 고정 + 테두리 하이라이트.
5. 4×2 위젯: 초기 스펙(카운트다운+게이지)에서 "주간 미리보기 그리드"로 변경. 근거 — 출근 전 카운트다운보다 주간 패턴을 한눈에 보는 것이 교대근무자의 주된 요구에 부합하며, 게이지 갱신의 OS 위젯 제약을 피할 수 있음.
6. 2×2 위젯: MVP 범위에서 제외, 추후 추가. 2×1·4×1 두 종으로 small/medium 사용 패턴을 충분히 커버.
7. 설정 화면: "앞으로 확인하지 않기" 재활성화 진입점을 별도 설정 화면(§4.4)으로 구현. 앱 정보(버전) 포함.

## 11. 데이터 수집 및 분석

### 11.1 원칙

- 식별자는 **앱 설치 시 생성하는 랜덤 UUID(익명)** 하나만 사용. 계정·이름·사번·광고ID·위치는 수집하지 않는다.
- **수집하지 않는 것(불변)**: 스케쥴 이미지(§6.3 동의 케이스 제외), 이미지에서 추출된 이름·사번. 진단 로그에 OCR 원문을 담을 때는 **6자리 이상 연속 숫자를 마스킹**(사번 방어)하고, 시간·근태코드 셀 영역의 텍스트만 담는다.
- 사용자의 스케쥴 시각 데이터(출퇴근 시간 값 자체)는 분석 서버로 보내지 않는다. 단, §11.3의 OCR 품질 진단에서 "파싱값 → 수정값" 쌍은 예외로 수집한다(시간 값은 민감정보가 아니며 OCR 개선의 핵심 데이터이기 때문).
- 모든 수집 항목은 개인정보처리방침과 스토어 데이터 라벨에 명시.

### 11.2 이벤트 정의 — 사용/리텐션

공통 속성: `anonymous_id, app_version, os, os_version, timestamp`

| 이벤트 | 속성 | 목적 |
|---|---|---|
| `app_open` | `source: icon \| widget_2x1 \| widget_4x1 \| widget_4x2_weekly` | **위젯 경유 유입 측정.** 위젯 탭 딥링크에 source 파라미터를 심어 계측. (widget_2x2는 MVP 제외이므로 없음) |
| `widget_active` | `types: []` — 앱 오픈 시 OS API로 현재 설치된 위젯 목록 조회 | 위젯 설치율 (iOS는 추가/제거 이벤트를 직접 못 받으므로 스냅샷 방식) |
| `home_week_viewed` | `offset` (0=이번주, -1=지난주…) | 과거/미래 스케쥴 열람 패턴 |
| `setting_changed` | `key: skip_confirm \| …, value` | 확인 생략 옵션 채택률 |

### 11.3 이벤트 정의 — 등록 퍼널 & OCR 품질 (핵심)

등록 1회마다 `session_id`로 묶어 퍼널 분석이 가능하게 한다.

| 이벤트 | 속성 | 목적 |
|---|---|---|
| `register_start` | — | 퍼널 시작 |
| `image_selected` | `image_width, image_height` | 화질 분포 파악 |
| `stage1_result` | `pass: bool, fail_reason: no_header \| no_date_pattern` | 비스케쥴 이미지 유입률 |
| `parse_result` | `failed_rows: n, duration_ms, ocr_confidence_avg` | 추출 성공률·성능 |
| `row_fail_detail` (실패 행마다 1건) | `row_index, fail_reason: NO_TEXT \| TIME_PARSE_FAIL \| INTEGRITY_FAIL \| ROW_NOT_FOUND, raw_text(마스킹 적용), cell_confidence` | **어떤 부분이 어떻게 실패했는지** — 이미지 없이 실패를 재구성하는 핵심 로그 |
| `send_dialog` | `consented: bool` | 실패 이미지 수집 동의율 |
| `confirm_shown` | `skipped: bool` | 확인 화면 노출/생략 |
| `user_edit` (수정 셀마다 1건) | `edit_source: confirm_screen` (현재는 confirm_screen 고정 — 향후 홈 수정 기능 추가 시 `home` 값 확장), `row_index, field: start \| end \| code, parsed_value, corrected_value, was_failed_row: bool` | **사용자가 무엇을 어떻게 고쳤는지** — 묵시적 실패(파싱은 성공했지만 값이 틀린 케이스) 탐지. 이게 OCR 정확도의 실측치. 단, 홈 수정은 OCR 오류가 아닌 실제 스케쥴 변경이므로 정확도 집계 시 `edit_source=confirm_screen`만 사용 |
| `register_complete` | `edited_rows, manual_rows, replace: bool, total_duration_ms` | 퍼널 완료·등록 소요시간 |
| `register_abandon` | `last_step` | 이탈 지점 |

> **묵시적 실패에 주목**: `parse_result`가 성공이어도 `user_edit`이 발생하면 OCR이 그럴듯한 오답을 낸 것이다(예: 15:00→16:00 오인식). "확인 생략" 사용자 비중이 늘기 전, 초기 사용자의 user_edit 데이터가 정확도를 검증할 거의 유일한 기회이므로 출시 초기 수집이 특히 중요하다.

### 11.4 수집 단계 (Phase)

| | Phase 1 — 출시 ~ 추출 정확도 검증 완료(목표 8주) | Phase 2 — 안정화 이후 |
|---|---|---|
| `row_fail_detail` | raw_text, confidence 포함 전체 수집 | fail_reason 카운트만 |
| `user_edit` | parsed/corrected 값 쌍 포함 | 수정 발생 여부·필드만 |
| 전환 방식 | 리모트 컨피그 플래그로 서버에서 전환 (앱 업데이트 불필요) | |

### 11.5 보관 및 도구

- 이벤트 원본 로그: 서버 보관 **180일**, 이후 집계 데이터만 유지. 실패 이미지는 §6.3대로 30일.
- 도구: Firebase Analytics(무료) + BigQuery 연동을 기본안으로 한다. `row_fail_detail`·`user_edit`처럼 속성이 많은 이벤트는 파라미터 제약을 고려해 설계 단계에서 스키마 검증 필요.
- §9 성공 지표와의 매핑: 위젯 설치율 ← `widget_active`, 재등록률 ← `register_complete` 주간 코호트, 추출 성공률 ← `parse_result` + `user_edit` 보정.
